01  ·  The Problem

Urban Flooding Demands Structured Decision Intelligence

Dense metropolitan areas face acute monsoon flooding across hundreds of administrative wards. Existing response mechanisms suffer from fragmented data pipelines, no reproducible advisory logic, and no structured citizen feedback loop — leaving officers to make high-stakes evacuation decisions without defensible evidence chains.

High rainfall variability with no automated ward-level classification
Delayed ground confirmation — SAR imagery often unavailable within response windows
No reproducible audit trail for evacuation advisories
Fragmented citizen reporting with no structured officer feedback loop
Manual review bottlenecks causing dangerous latency in life-safety decisions
"Flood advisories must be fast, sensitive, and legally defensible — simultaneously. Current systems cannot achieve all three."

03  ·  Intelligence Design

05  ·  Governance Workflow

Human-in-the-Loop Enforcement — By Design, Not Policy

No public evacuation advisory is published without a duty officer sign-off. Double-submission is blocked at the API level per run ID. Every decision is immutably stored with officer identity, role, timestamp, and rationale.

09  ·  Governance & Compliance

Audit-Ready by Design

The platform treats institutional accountability as a first-class design requirement — not an afterthought. Every architectural decision, from run isolation to decision locking, is built to satisfy an auditor, not just an operator.

🔐
Role-Based Access Control
Only authenticated Duty Officers can submit human decisions. Session tokens validated server-side. Guests see scores but cannot write decisions.
🔒
Immutable Run Separation
Each cycle generates a unique run_id. Decisions are locked per run — no retroactive modification. A new run creates a clean, isolated audit state.
🚫
Double-Submit Prevention
Before any submission, system queries existing decisions for that ward in the current run. If found, action is fully blocked with prior decision shown.
📊
KPI-Based FP Management
Target Detection ≥85%, FP ≤10%. If FP exceeds threshold in calibration, automated publishing is frozen per SOP-X §13.
🔏
SHA256 Decision Packets
Auto-generated per run: GPM inputs, SAR product IDs, composite breakdown, officer sign-off. Immutably retained for 24hr auditor access.
📡
SAR Outage Fallback Mode
When Sentinel-1 unavailable, advisory requires Composite ≥85 + one static HIGH + officer sign-off with field confirmation. Labeled SAR_UNAVAILABLE_MODE.

12  ·  Operational Constraints & Risk Controls

Known Constraints & Mitigation Design

Responsible systems design requires explicit constraint documentation. The following operational limitations are acknowledged, with architectural risk controls identified for each.

SAR Revisit Dependency — Sentinel-1 confirmation is subject to satellite revisit frequency and acquisition timing, which may delay independent flood verification within narrow response windows. Fallback mode (SOP-X §6) activates automatically.
Proxy Label Precision — Proxy-labelled historical events may influence ML precision where SAR-confirmed ground truth is unavailable. Temporal fold analysis flags affected calibration years for recalibration review.
Rainfall Interpolation Accuracy — Rainfall aggregation accuracy depends on GPM spatial resolution and ward-level interpolation methodology. Coarse rainfall bins are intentional to reduce classification sensitivity at uncertain boundaries.
Field Verification Requirement — Field verification remains mandatory for critical infrastructure evacuation decisions, regardless of composite score. Officer sign-off architecture enforces this by design, not policy.
Seasonal Threshold Drift — Threshold sensitivity must be seasonally recalibrated to avoid over-triggering in anomalous rainfall clusters. A/B historical re-run is required before each monsoon seasonal lock per SOP-X §1.
Static Layer Vintage — Terrain, built-up, and hydraulic layers are version-locked per season. Significant urban morphology changes require layer updates and DMU-approved change control per SOP-X §17.
